ANÁLISE COMPLETA DO PRODUTO SLM MEDTEACHER                                                                                                                                                                                                  
   
  1. O QUE FAZ ESTE PRODUTO                                                                                                                                                                                                                   
                                                                     
  O SLM MedTeacher é um assistente clínico de inteligência artificial que roda localmente no computador do profissional de saúde, com opção de fallback para a nuvem. Seu propósito é auxiliar na interpretação de exames médicos e no
  raciocínio clínico estruturado.

  Capacidades Funcionais

  ┌───────────────────────────┬──────────────────────────────────────────────────────────────────────────┐
  │          Recurso          │                             Tecnologia Real                              │
  ├───────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ Chat clínico textual      │ MedGemma 4B/27B (local, Ollama) ou Gemini 2.5 Flash (nuvem)              │
  ├───────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ Análise de ECG por imagem │ Triple ensemble: ConvNeXt (BHF) + HuBERT + QueenBee + Ahus-AIM digitizer │
  ├───────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ Análise de Raio-X/TC      │ TorchXRayVision DenseNet121 (classificação multi-label)                  │
  ├───────────────────────────┼──────────────────────────────────────────────────────────────────────────┤

  ├───────────────────────────┼──────────────────────────────────────────────────────────────────────────┤

  Stack Tecnológica Detalhada

  - Servidor: FastAPI 0.115 + Uvicorn
  - LLMs: Ollama (MedGemma 4B/27B quantizado Q4_K_M) ou Google Gemini 2.5 Flash via API REST
  - Visão Computacional: PyTorch, TIMM (ConvNeXt), Transformers (HuBERT, QueenBee), TorchXRayVision, OpenCV
  - ECG Digitization: Ahus-AIM (UNet segmentação, Hough Transform, Thin Plate Spline, Hungarian Algorithm)
  - OCR: PaddleOCR v3.4+
  - Privacidade: Microsoft Presidio + SpaCy (pt_core_news_lg, en_core_web_lg)
  - Frontend: Vanilla HTML5/JS/CSS, sem framework

  2. PIPELINE DE ANÁLISE DE ECG — DESCRIÇÃO DETALHADA

  3.2. Por Que _extract_signal() Retorna None

  Existem 3 cenários mutuamente exclusivos:

  Cenário A: self.digitizer nunca foi criado (mais provável)

  Ponto no código: ecg_analyzer.py:149
  def _extract_signal(self, image_bytes):
      if not self.digitizer: return None   # ← self.digitizer é None

  Razão 1 — Falha de importação (ecg_analyzer.py:24-27):
  try:
      from src.model.inference_wrapper import InferenceWrapper
  except ImportError:
      InferenceWrapper = None  # ← se QUALQUER import transitivo falhar

  O InferenceWrapper importa src.utils, que por sua vez importa:
  - ray.tune.Stopper — RAY NÃO ESTÁ NO requirements.txt
  - networkx (usado em dewarper.py) — NÃO ESTÁ NO requirements.txt
  - sklearn.neighbors.NearestNeighbors — NÃO ESTÁ NO requirements.txt
  - torch_tps.ThinPlateSpline — NÃO ESTÁ NO requirements.txt
  - matplotlib — NÃO ESTÁ NO requirements.txt
  - timm — NÃO ESTÁ NO requirements.txt
  - presidio_analyzer, presidio_anonymizer — NÃO ESTÃO NO requirements.txt
  - torchxrayvision — NÃO ESTÁ NO requirements.txt

  A cadeia de importação é:
  1. ecg_analyzer.py → from src.model.inference_wrapper import InferenceWrapper
  2. inference_wrapper.py → from src.utils import import_class_from_path
  3. src/utils.py → from ray.tune import Stopper ← ray não instalado = ImportError

  Mesmo que ray estivesse instalado, quando _init_digitizer() tenta instanciar os sub-módulos, eles importam:
  - dewarper.py → import networkx as nx + from torch_tps import ThinPlateSpline + from sklearn.neighbors import NearestNeighbors
  - signal_extractor.py → import matplotlib.pyplot as plt

  Razão 2 — Exceção silenciosa em _init_digitizer() (ecg_analyzer.py:64-68):
  if InferenceWrapper:
      try:
          self._init_digitizer()
      except Exception as e:
          logger.error(f"Erro ao configurar Digitizer: {e}")
          # self.digitizer permanece None — erro engolido silenciosamente

  Causas possíveis de exceção em _init_digitizer():
  - Arquivo de pesos ecg_segmentation_unet.pt não encontrado ou corrompido
  - Arquivo de pesos lead_name_unet_weights_07072025.pt não encontrado
  - Incompatibilidade de checkpoint (keys com _orig_mod. prefix)
  - YAML config files ausentes

  Cenário B: Digitizer funciona mas canonical_lines é None

  Ponto no código: ecg_analyzer.py:155-156
  canonical = got_values.get("signal", {}).get("canonical_lines")
  return canonical.cpu().numpy() if canonical is not None else None

  Isto ocorre quando LeadIdentifier.__call__() retorna canonical_lines = None — o que acontece apenas se len(detected) <= 2 (lead_identifier.py:478-480):
  if len(detected) <= 2:
      match = {"cost": float("inf")}
      canonical_lines = None  # ← definido mas NUNCA usado

  Porém, olhando o código mais cuidadosamente (lead_identifier.py:484-494):
  if "layout" not in match:
      match["layout"] = list(layouts.keys())[0]  # fallback
  canonical_lines = self._canonicalize_lines(lines.clone(), match)
  return {
      ...
      "canonical_lines": canonical_lines,  # ← sempre retorna algo (pode ter NaN)
  }

  O LeadIdentifier na verdade sempre retorna canonical_lines (mesmo que cheio de NaN). Então este cenário é menos provável do que o Cenário A, a não ser que layouts esteja vazio e list(layouts.keys())[0] cause IndexError.

  Cenário C: O digitizer levanta exceção durante forward()

  Se InferenceWrapper.forward() levantar qualquer exceção (tensor com dimensões inesperadas, divisão por zero no PixelSizeFinder, linhas paralelas no Cropper, etc.), ela propagaria para _extract_signal() e seria capturada pelo try/except
  genérico em analyze() (ecg_analyzer.py:108-110), que retornaria {"error": str(e)} — uma mensagem diferente da observada.

  Porém, como o _extract_signal() não tem try/except interno, se a exceção ocorre lá dentro, ela vai direto para o except do analyze(), resultando em uma mensagem de erro diferente (o str(e) da exceção real). Isso significa que o erro
  "Falha na extração de sinais. Pipeline incompleta." só ocorre quando self.digitizer é None.

  def _estimate_hr(self, signal_np):
      if signal_np is None:
          return None
      # Usar lead II (índice 1) para detecção de picos R
      lead_ii = signal_np[1] if signal_np.shape[0] > 1 else signal_np[0]
      lead_ii = lead_ii[~np.isnan(lead_ii)]
      if len(lead_ii) < 100:
          return None
      from scipy.signal import find_peaks
      peaks, _ = find_peaks(lead_ii, distance=len(lead_ii)//20, prominence=np.std(lead_ii)*0.5)
      if len(peaks) < 2:
          return None
      rr_intervals = np.diff(peaks)
      # 5000 amostras = 10 segundos (padrão ECG) → sample_rate = 500 Hz
      sample_rate = 500
      mean_rr = np.mean(rr_intervals) / sample_rate
      return int(60 / mean_rr) if mean_rr > 0 else None
